# Code Review — Accumulated Knowledge

> Living notes for `/code-review` + `code-reviewer`: recurring smells and judgment calls learned with use.
> Authority for the anti-pattern codes stays in `.claude/patterns/code-review-checklist.md`.

## Recurring findings

- **Cache-aside + soft-delete interaction:** when a repository's `findById` does not filter `deleted_at: null` at the Prisma level (user repo does `findUnique` without the deleted-at filter), a soft-deleted entity can be cached and returned as live on subsequent hits. Always verify whether the underlying Prisma query includes the soft-delete filter; if not, add a post-deserialize guard or fix the Prisma query.
- **Write-then-evict ordering:** every `inner.write → await invalidateCache` pair has a sub-TTL stale window for concurrent readers. This is inherent to write-aside invalidation; document the worst-case window (= TTL) in env comments, especially for security-adjacent paths (webhook URLs, token data).
- **`container.resolve()` per-request in middleware:** resolving singletons from the DI container on every request is a performance and consistency smell. Capture singletons in the factory-function closure at middleware creation time.
- **`InMemoryCacheProvider.increment` ignores TTL:** the in-memory mock stores counters indefinitely, diverging from real Redis which applies a TTL on first insertion. Security-gate tests that rely on TTL expiry will produce false-passing results.

## Judgment calls / calibration

- The confused-deputy risk of unscoped-cache + in-memory `accountId` guard (`CachedDevicesRepository.findById`) is sound when Redis is a trusted internal boundary and not shared across tenants or apps. The trust assumption must be named at the method level as an auditable invariant.
- Full-parity codec (including `password` bcrypt hash and `refreshTokenHash`) in `cached-user-repository.ts` is intentional and correct: nulling credentials on a cache hit would create hit≠miss divergence. The rationale should be documented at the codec (already is, as of this review).
- `signUpTransaction` not evicting is correct because the new user id cannot be in the cache. However, if that method is ever changed to update an existing user, the no-evict assumption silently breaks — add a comment.

- **Dead-type select in sandbox forms:** when a SelectField exposes message-type options but the send handler ignores the selected value and always calls the text endpoint, the select is visually meaningful but functionally dead. Either remove it (until a second type exists) or gate `handleSend` on `formData.messageType` routing. The pattern recurs any time spec §x says "text only, type selector is for future use."
- **Stale-closure in device-select useEffect:** `useFormState.setField` depends on `formData` and recreates on every form change. A `useEffect(() => { ... }, [connectedDevices])` that calls `setField` and also reads `formData.deviceId` inside the closure will capture a stale `formData` if the user changed the select between two `connectedDevices` updates. Fix: either include `form.formData.deviceId` in the dep array (may cause spurious re-runs) or store the selected id in a separate `useRef` that is always current.
- **`updateStatus` unscoped-by-account in DisconnectDeviceUseCase:** this is intentional and documented in `IDevicesRepository` — the ownership check happens at `findById(accountId, id)` before `updateStatus(device.id, ...)` is called. The pattern is safe as long as `device.id` comes from a prior scoped read. Never forward `device.accountId` from an unscoped method into a scoped query (confused-deputy, per the interface comment).

- **Shadow token rename drift:** when semantic-tokens.ts renames shadow tokens (e.g. `card-hover` → `shadow.cardHover`), grep ALL component files for the old token string before closing the task. Chakra silently ignores unknown `boxShadow` values — no build error, no TS error, only a visual regression. The check `grep -rn '"card-hover"' apps/web/src --include="*.tsx"` must pass clean after any shadow rename.
- **Polling query: write-before-send contract verification:** when a FE polling hook fires `GET /resource/:id` after a `POST` that returns the id, confirm the backend `POST` handler writes the row to the DB BEFORE it attempts the side-effect (gateway call) — otherwise the first poll arrives before the row exists and gets a 404. In `SendTextMessageUseCase`, `outboxRepository.create()` is called at line 95 and `gateway.sendText()` at line 121; the 202 response carries the committed id, so `GET /messages/:id` is guaranteed to find the row. Document this ordering as an invariant when reviewing messaging use cases.
- **MobileBottomNav item duplication:** `MobileBottomNav.tsx` maintains its own flat array of nav items in parallel to `navigation.ts` (which is the authority for the sidebar). Adding a nav item requires editing both files. Prefer extending `NavigationItem` with an optional `matchPaths?: string[]` field so both surfaces draw from the single `navigationSections` source. Flag any PR that adds items to one and not the other.
- **`useColorMode` in SidebarNav is not F-H16:** using `colorMode === "dark"` to toggle icons (FiSun/FiMoon) or translate strings ("Switch to dark/light theme") is a legitimate, non-semantic-token use of `useColorMode`. F-H16 only applies to color value selection — `isDark ? "red.600" : "blue.600"` — not to icon or text selection. Do not flag the SidebarNav color-mode toggle as F-H16.

- **Concrete-class injection without @inject in tsyringe:** when a constructor parameter type is a concrete `@injectable()` class (not an interface), tsyringe resolves it via `reflect-metadata` type token without requiring `@inject(TOKEN)`. This works today because `emitDecoratorMetadata: true` emits the class constructor as the design:paramtypes entry. It breaks if: (a) the param type is imported as `type` only (erased at runtime), (b) a second impl of the same interface replaces the class, or (c) the class is moved to a different module and the singleton registration is missed. The canonical pattern for this project is to always use `@inject(DI_TOKENS.X)` for every parameter — even for concrete class tokens — so the wiring is explicit and survives refactors.

- **waitForToken spin-safety in drain loop:** the drain's `waitForToken` private method avoids busy-spin by sleeping `Math.max(msUntilNextToken(), 1)` ms between retries. The `unref()` on the internal `setTimeout` prevents the timer from blocking graceful shutdown. Exit is conditional on `isConnected` re-checked inside the loop, so a device that disconnects while waiting returns `false` promptly (no indefinite hold on the single-flight guard). The pattern is sound; the only risk is the drain holding the guard indefinitely if the device reconnects faster than the token refill — but that maps to a legitimate live-drain, not a starved session.

- **Token bucket `msUntilNextToken` called inside `waitForToken` after `tryConsume` returned false:** calling `msUntilNextToken` right after `tryConsume` returned false means the refill math re-runs (`bucketFor`) but the elapsed time since the last consume is near zero. The computed sleep is therefore accurate (it is `ceil((1 - tokens) / refillPerMs)` where `tokens` is already negative of one token). No double-consume risk because `msUntilNextToken` is a read-only path; only `tryConsume` mutates `bucket.tokens`.
