🕊️ Pombo — como o projeto funciona

Um gateway de WhatsApp não-oficial: transforma um número de WhatsApp comum num serviço com API REST + webhooks.
Node.js + Express 4 TypeScript Baileys (WhatsApp Web) Prisma 7 + Postgres tsyringe (DI) Clean Architecture Monorepo Turborepo

00Resumo em 1 minuto

O Pombo pega um número de WhatsApp comum (um chip), faz o mesmo que o WhatsApp Web faz — se conecta como se fosse um "aparelho vinculado" — e expõe isso como um serviço: você registra um dispositivo, escaneia um QR Code uma vez, e a partir daí manda mensagens via API REST e recebe atualizações via webhook.

Analogia: pense no Pombo como um funcionário que segura o seu celular. Uma vez que você mostra o QR Code pra ele (o vínculo), ele consegue mandar mensagens pelo seu número quando você pedir (API), e te avisa por bilhete (webhook) toda vez que algo acontece — "mensagem entregue", "mensagem lida", "o celular caiu da rede". Ele nunca inventa nada: o WhatsApp é sempre a fonte da verdade, o Pombo só traduz.

As quatro perguntas que este documento responde:

  • Como conectamos ao WhatsApp e recebemos o QR Code? → §03, §04
  • Como recebemos eventos do WhatsApp? → §05 (um event bus interno)
  • Como mandamos webhooks? → §09 (POST assinado com HMAC, com retry)
  • O que é uma sessão / vínculo de dispositivo? → §02, §06

02Glossário — os conceitos, sem jargão

Antes dos fluxos, cinco palavras que aparecem o tempo todo. Vale fixar cada uma:

📱 Device (Dispositivo)

A unidade central. É um registro no banco que representa "um chip de WhatsApp que o Pombo gerencia". Tem um name, um webhook_url (pra onde mandar os avisos), um webhook_secret (pra assinar esses avisos) e um status. Ele existe no banco antes de qualquer conexão — registrar e conectar são passos separados.

Device ≠ conexão. Registrar um device é como criar a ficha do funcionário. Conectar (escanear o QR) é o funcionário bater o ponto. Você pode ter a ficha sem ninguém trabalhando naquele momento.

🔗 Session (Sessão)

É a conexão viva entre o Pombo e os servidores do WhatsApp para um device — um WebSocket aberto, falando o protocolo do WhatsApp. Na prática é um objeto WASocket do Baileys, guardado em memória num Map<deviceId, socket>. Enquanto a sessão está "open", você consegue mandar mensagem. Se cai, o Pombo tenta reabrir sozinho. Sessão é volátil (vive em memória); o que a torna reconstruível é o AuthState 👇.

🔑 AuthState (o vínculo persistido)

Aqui mora a mágica do "não precisa escanear o QR toda vez". Quando você pareia um device, o WhatsApp e o Baileys negociam um monte de chaves criptográficas do Signal Protocol. Isso não é uma senha — é um estado criptográfico que muda a cada mensagem. O Pombo persiste esse estado na tabela auth_key (uma linha por chave, serializada com BufferJSON pra os bytes sobreviverem).

AuthState é o "vínculo do aparelho". É exatamente aquela linha em WhatsApp > Aparelhos conectados no seu celular. Enquanto ela existe (no celular e no banco do Pombo), o Pombo reconecta sozinho depois de um deploy ou reboot — zero QR. Quando você faz logout, esse vínculo é apagado dos dois lados e aí sim precisa parear de novo.

📮 Outbox (caixa de saída)

Uma tabela onde toda mensagem enviada é gravada antes de ir pro WhatsApp. Não é um "histórico de conversa" — é protocolo. Existe por dois motivos concretos:

  • Idempotência: a coluna única (device_id, idempotency_key) garante que, se o seu sistema mandar o mesmo request duas vezes (timeout, retry), a mensagem sai uma vez só.
  • Reenvio (getMessage): se o destinatário não conseguir descriptografar, o WhatsApp pede o texto original de volta — o Baileys busca no outbox.

Por isso ela é podada por TTL (expires_at): depois de ~24h a linha já cumpriu seu papel e é apagada.

📨 Webhook

O caminho de volta. Quando algo acontece (device conectou, mensagem foi lida...), o Pombo faz um POST pro webhook_url que você configurou, assinado com HMAC-SHA256 usando o webhook_secret. Você valida a assinatura e confia que veio mesmo do Pombo.

03A conexão com o WhatsApp — o que é o Baileys

O WhatsApp não tem uma API oficial gratuita pra número comum. O que existe oficialmente é o WhatsApp Web — aquele QR Code que vincula seu navegador ao celular. O Baileys (@whiskeysockets/baileys) é uma biblioteca open-source que fala esse mesmo protocolo do WhatsApp Web, direto por WebSocket, sem navegador. O Pombo é, no fundo, um WhatsApp Web headless e programável.

Baileys fica isolado atrás de uma "porta". Só três arquivos no projeto inteiro conhecem tipos do Baileys: session-manager.ts, socket-config.ts e prisma-auth-state.ts. Todo o resto do código conversa com a interface IWhatsAppGateway (a "porta"). Isso é o padrão Ports & Adapters.

Existem três implementações dessa mesma porta, escolhidas na inicialização:

ImplementaçãoQuando é usadaComportamento
BaileysWhatsAppGatewayWHATSAPP_ENABLED=trueA real. Abre sockets de verdade. O Baileys só é carregado (via import() dinâmico) aqui.
DisabledWhatsAppGatewayWHATSAPP_ENABLED=false (padrão)No-op. A API sobe normal, mas connect() falha limpo com WA_GATEWAY_DISABLED. Baileys nunca carrega.
FakeWhatsAppGatewaytestesSimula tudo em memória — 100% dos use-cases testáveis sem socket real.

A porta em si é enxuta — é só isto:

interface IWhatsAppGateway {
  connect(deviceId)      // abre a sessão (dispara o QR se preciso)
  disconnect(deviceId)   // fecha o socket, MAS mantém o vínculo
  logout(deviceId)       // desfaz o vínculo (apaga o pareamento)
  isConnected(deviceId)  // a sessão está "open"?
  resolveJid(id, phone)  // "esse número existe no WhatsApp?"
  sendText(id, jid, txt) // manda o texto
  isEnabled()            // o gateway está ligado?
}
      apps/api/src/modules/devices/domain/provider/whatsapp-gateway.interface.ts
disconnect ≠ logout. Guarde essa diferença, ela é o coração de tudo:
  • disconnect() / closeAll() → fecha o socket mas preserva o AuthState. É o que roda no deploy. Reconecta sem QR.
  • logout() → desfaz o vínculo no WhatsApp e apaga o AuthState. É o DELETE /devices/:id. Precisa de QR novo depois.

O QR aparece no terminal do servidor (via qrcode-terminal) — é a UX de pareamento sancionada. O identifier (o número real) só é conhecido depois do "open", porque quem manda nisso é o WhatsApp.

De onde vem o QR Code, tecnicamente

Quando o socket abre sem um vínculo válido, o WhatsApp manda um evento connection.update com um campo qr (uma string). O session-manager.ts pega isso e publica no event bus como session.qr: 

sock.ev.on("connection.update", (update) => {
  const { connection, lastDisconnect, qr } = update;
  if (qr) deps.bus.publish({ type: "session.qr", deviceId, qr });
  // ...
});
      session-manager.ts — a tradução do socket em evento de domínio

Quem consome o session.qr é o boot do gateway, que imprime o QR bonitinho no terminal. O QR é rotativo: se você não escanear a tempo, o WhatsApp manda outro, e o Pombo reimprime. Note que o QR não vira webhook — ele é efêmero e serve só pro pareamento.

Por que registrar e conectar são separados? Porque a ficha do device (e o webhook_secret) precisa existir antes de qualquer socket — e porque conectar é re-executável: se o device deslogar, você chama /connect de novo pra pegar um QR novo, sem recriar a ficha nem perder a configuração de webhook.

05Como recebemos eventos — o Domain Event Bus

Esta é a peça de arquitetura mais importante pra "entender como funciona". O WhatsApp é assíncrono: a qualquer momento chega um "mensagem entregue", "conexão caiu", "QR novo". O Baileys entrega tudo isso num stream de eventos (sock.ev). O problema: quem reage a cada coisa? O messaging quer saber de status; o webhooks quer saber de tudo; o devices quer atualizar o próprio status.

A solução é um event bus interno e tipado (IDomainEventBus) — um publish/subscribe em processo. O adapter Baileys traduz os eventos crus do socket em eventos de domínio e publica. Os módulos assinam o que lhes interessa. Assim devices nunca importa webhooks: eles só se encontram no bus.

As três reações, e por que a ordem importa

EventoUse-caseO que faz
session.connectedHandleSessionConnectedUseCasestatus → CONNECTED, grava o identifier (o número, que só o WhatsApp sabe) e carimba lastConnectedAt.
session.disconnectedHandleSessionDisconnectedUseCasestatus → DISCONNECTED. É informativo ("caí, já volto"). A camada de socket reconecta sozinha com backoff exponencial.
session.logged_outHandleSessionLoggedOutUseCasestatus → LOGGED_OUT e depois apaga o AuthState.
Por que status primeiro, AuthState depois? As duas escritas não são transacionais. Se apagar o AuthState primeiro e o status falhar, você fica com o pior estado possível: vínculo apagado mas /health dizendo "CONNECTED" (mente). Fazendo status primeiro, se o segundo passo falhar, sobra só chaves obsoletas e inofensivas (não há reconexão em LOGGED_OUT), que se auto-curam no próximo /connect manual.

Reconexão automática (o backoff)

Quando cai (e não foi logout), o session-manager agenda um retry com atraso exponencial: base × 2^tentativa, com teto. Configurável por RECONNECT_BASE_DELAY_MS (3s) e RECONNECT_MAX_DELAY_MS (5min). O timer é unref()ado — assim um reconnect pendente nunca segura o processo no shutdown.

A gravação no outbox acontece antes do sendText — porque se o WhatsApp pedir reenvio, o texto precisa já estar salvo.

Idempotência — a garantia contra "mandou duas vezes"

A coluna única (device_id, idempotency_key) no Postgres é a guarda real. O use-case ainda tem um fast-path em memória (se a chave já existe com o mesmo texto, faz replay do 202 sem reenviar), mas mesmo numa corrida concorrente, o banco garante que só uma linha vence — a perdedora relê a vencedora e retorna o mesmo messageId.

Idempotency-Key é o número do pedido. Se o garçom não ouviu você confirmar e você repete "mesa 3, um chopp" com o mesmo número de comanda, sai um chopp — não dois. Texto diferente com a mesma comanda? Aí ele acusa o conflito (409): comanda já usada.

Por que o outbox não é histórico

Ele é protocolo, não memória. Cumpre idempotência e reenvio, e depois é descartável. Um job (PruneOutboxJob) roda a cada hora (OUTBOX_PRUNE_INTERVAL_MS) e apaga tudo com expires_at < agora — o TTL padrão é 24h (OUTBOX_TTL_HOURS). Se você quer histórico de conversa de verdade, é outra feature.

08Status monotônico — o pulo do gato da entrega

O WhatsApp manda ACKs de entrega, mas eles podem chegar fora de ordem (a rede não garante ordem). Sem cuidado, você veria "lida" e depois "entregue", uma regressão que confunde qualquer sistema. A solução é tratar o status como monotônico: ele só pode subir de rank, nunca descer.

    direction LR
    PENDING --> SERVER_ACK: ✓ (rank 0→1)
    SERVER_ACK --> DELIVERY_ACK: ✓✓ (1→2)
    DELIVERY_ACK --> READ: ✓✓ azul (2→3)
    PENDING --> FAILED
    SERVER_ACK --> FAILED
    DELIVERY_ACK --> FAILED
    READ --> READ: 🔒 nada regride READ
    note right of READ
        RANK: PENDING 0 · SERVER_ACK 1
        DELIVERY_ACK 2 · READ 3 · FAILED 4
        Regra: só sobe de rank.
        FAILED é permitido de tudo,
        EXCETO de READ.
    end note
A transição só é aceita se o rank de destino for maior — exceto FAILED, que pode vir de qualquer estado, menos de READ (uma mensagem lida não "falha").

A regra em código é literalmente uma linha:

canTransitionTo(from, to) =>
  to === "FAILED" ? from !== "READ" : RANK[to] > RANK[from]
      messaging/domain/value-object/message-status.ts

Onde a guarda é aplicada: no WHERE do banco

O mais elegante é que a guarda não é um if em memória — ela vive na cláusula WHERE do UPDATE, o que a torna atômica (imune a corridas concorrentes): 

UPDATE outbox_message
   SET status = :novoStatus
 WHERE wa_message_id = :id
   AND status IN (:estadosDosQuaisÉPermitidoSubir)

Se o UPDATE não casar nenhuma linha (ACK atrasado, regressivo, ou desconhecido), é um no-op silencioso. Só quando há uma subida real é que o messaging re-publica o evento message.status (agora "de negócio", com o messageId do outbox, não o waMessageId cru) — e é esse evento que o webhook consome.

Por isso o webhook nunca regride e o módulo webhooks nunca precisa tocar no outbox: a separação session.message_status (cru) → message.status (negócio) garante que o cliente só é notificado de progresso real e monotônico.

O debouncer na desconexão é o que evita "webhook-spam" quando a rede pisca: um connected ou logged_out chegando dentro da janela cancela o device.disconnected pendente.

O envelope e a assinatura

Todo webhook tem o mesmo formato de corpo:

{
  "type": "message.status",
  "deviceId": "d1",
  "eventId": "evt_<uuid>",
  "timestamp": "2026-07-17T12:00:00.000Z",
  "data": { "messageId": "...", "status": "READ" }
}

E vai com estes headers. A assinatura cobre timestamp . corpo, então adulterar qualquer um quebra a validação (anti-replay embutido):

HeaderConteúdo
X-Signaturesha256=<hmac hex> — HMAC-SHA256 de `${timestamp}.${rawBody}` com o webhook_secret
X-TimestampUnix em segundos (o mesmo instante do campo timestamp)
X-Event-Idevt_<uuid> — pra você deduplicar do seu lado
const mac = createHmac("sha256", secret)
  .update(`${timestamp}.${rawBody}`)
  .digest("hex");
return `sha256=${mac}`;
      webhooks/infrastructure/provider/hmac-signer.ts
Como você valida do seu lado: pega o corpo cru recebido, junta com o X-Timestamp, calcula o mesmo HMAC-SHA256 com o segredo que você guardou no registro, e compara com o X-Signature. Bateu → veio do Pombo. É o mesmo esquema do Stripe/GitHub.

Retry & timeout

  • Retenta em 5xx, 429 e timeouts/erros de rede (at-least-once).
  • Não retenta em 4xx (erro do cliente — insistir não adianta).
  • Backoff exponencial WEBHOOK_RETRY_BASE_DELAY_MS × 2^(n-1), até WEBHOOK_MAX_ATTEMPTS (4) tentativas.
  • Timeout de WEBHOOK_TIMEOUT_MS (5s) por tentativa, via AbortController.
  • Falhou tudo? Loga um warning e desiste — nunca derruba nada. O estado autoritativo continua disponível via GET /api/devices.

10Boot, advisory lock & shutdown gracioso

Quando a API sobe com WHATSAPP_ENABLED=true, uma sequência cuidadosa acontece pra garantir exatamente uma réplica segurando os sockets e pra reconectar sozinho os devices que estavam vivos — sem QR. 

flowchart TB

gateway desligado"]
    ENABLED -->|true| LOCK["🔒 pega o Postgres
advisory lock"]

    LOCK -->|ok| WIRE["liga os listeners do bus:
session · message · webhook"]

(CONNECTING/QR_PENDING/CONNECTED)

    RESET --> REHYDRATE["🔁 rehydration:
reabre quem estava CONNECTED
a partir do AuthState"]
    REHYDRATE --> PRUNE["inicia o prune do outbox (TTL)"]
    PRUNE --> LISTEN2["API escuta na porta ✅"]

    LISTEN2 -.->|SIGTERM| SHUT["shutdown gracioso"]

SEM logout (preserva vínculo)"]
    S1 --> S2["para o prune"]
    S2 --> S3["🔓 libera o advisory lock"]
A rehydration é o que faz deploy ser um não-evento: ninguém escaneia nada, os sockets voltam do AuthState salvo. O shutdown fecha com close(), nunca logout() — senão o deploy desfaria o pareamento.

O advisory lock — por que "exatamente uma réplica"

O AuthState é estado do Signal Protocol que muta a cada mensagem. Se duas réplicas do Pombo abrissem sockets com o mesmo AuthState, as chaves Signal corromperiam — a sessão morre. Por isso, antes de abrir qualquer socket, o boot adquire um Postgres advisory lock (pg_try_advisory_lock) numa conexão dedicada. Quem não conseguir o lock, exit(1) — se recusa a rodar. Um heartbeat vigia a conexão; se ela cair, o processo se mata pra ser reiniciado. É a garantia de réplica única.

11Garantias & segurança — o que torna isso confiável

🔒 Réplica única

Advisory lock no Postgres impede dois processos corromperem o AuthState. Heartbeat + exit(1) se o lock cair.

🔁 Deploy sem re-pareamento

AuthState persistido + rehydration + close() (nunca logout()) no shutdown → sockets voltam sozinhos.

♻️ Idempotência real

(device_id, idempotency_key) único no banco: retry do cliente nunca duplica mensagem.

📈 Status que nunca regride

Guarda monotônica na cláusula WHERE (atômica) + separação evento cru/negócio.

🔏 Webhooks à prova de forjaria

HMAC-SHA256 sobre timestamp.body, anti-replay embutido. Retry só em falhas transitórias.

🚪 Autenticação do painel

Todas as rotas de device/mensagem exigem JWT (authMiddleware). Helmet, CORS allowlist, CSRF, rate-limit no core/http.

🧪 100% testável sem socket

FakeWhatsAppGateway troca o Baileys nos testes — a lógica de negócio roda toda sem WhatsApp real.

🩺 Health honesto

GET /api/health só diz "ok" se todo device estiver CONNECTED; qualquer degradação → 503.

O fio condutor do design: o WhatsApp é sempre a fonte da verdade; o Pombo só reflete esse estado com fidelidade. Cada decisão (ordem das escritas, lock, monotonicidade, assinatura) existe pra que o banco nunca "minta" sobre o que o WhatsApp realmente fez.
